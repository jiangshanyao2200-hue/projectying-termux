# 2026-05-23 SSE timeout / retry 全链路修复
- 默认 API timeout 已统一改为 240s；当前 `config/{matrix,coding,server,advisor}` 与 `config/lastgood/*` 里的旧 60s 也同步更新，避免恢复 lastgood 后退回短超时。
- SSE timeout 语义改为 transport idle：用户设置的 timeout 对流式请求表示“连续无任何 stream bytes 的最长等待”，不是整次请求总时长，也不是“无模型语义 delta”。非 SSE 仍按一次完整请求总时长处理。
- 关键修复：SSE client 现在同时设置 `connect_timeout(timeout)` 和 `timeout(timeout)`。reqwest blocking 会把 `timeout()` 传给 `Response::read()`，每次底层 read 单独计时；只要服务端持续发送 SSE 字节或心跳，就不会因为总时长被误杀，真正停流才进入重试。
- SSE reader 现在在读到任意行字节时刷新 stream activity，包括 heartbeat/comment 行；旧的 `semantic idle timeout after 60s without model progress` 逻辑已废弃。
- 重试上限统一为 5 次重连；最终错误统一写 `连续重连失败：5/5` + `重试明细`，包含初始尝试和每次重试，前台 system error 展开后应能看到完整 ledger。
- Codex provider 不再强制 SSE；API 设置里的 SSE 开关现在对 Codex 也生效。SSE off 会走非流式请求，是否被上游接受由用户自行判断。
- 已补本地 TCP 回归：服务端返回 `text/event-stream` headers 后不发 body，客户端必须在 provider idle timeout 内结束读等待，而不是等服务端关闭。
