# ProjectYing 项目蓝图

本文件定义 `ProjectYing` 的目标结构与职责边界。

- `INDEX.md` 回答“现在代码和目录长什么样”。
- `项目记忆.md` 记录“历史上做过什么、为什么这么做”。
- 本蓝图只回答“接下来系统应该如何分工、哪些边界必须收紧”。

当前项目已经进入“先理清边界，再持续收口”的阶段，因此后续结构整改、工具裁剪、目录搬迁，都以本蓝图为准。

---

## 1. 总体原则

### 1.1 城市化原则

- 保持单文件主城思路，但每一块能力都要有明确主责城区。
- 一项能力只认一个主责方；其它模块只能调用，不再私造平行实现。
- 功能先能跑，再回归收口，但最终必须回到统一结构，避免长期拼接。

### 1.2 分层原则

ProjectYing 后续固定为三层：

1. **前台层**
   - 面向用户直接说话、接任务、汇报结果。
   - 代表：`Matrix`、`Coding`、`Companion`。
2. **执行层**
   - 真正做工具调用、代码修改、终端执行、读写文件。
   - 代表：`mcp.rs`、Provider 工具轮、PTY/agent 执行链。
3. **治理层**
   - 负责上下文治理、长期记忆维护、审计建议、日志观测、回归证据。
   - 代表：`Advisor`、`departmentrs.rs`、`memory/`、系统维护票据。

### 1.3 存储原则

- `config/` 只放配置，不混日志和缓存。
- `context/` 只放各 persona 的外置上下文与 schema。
- `memory/` 只放长期记忆与工具历史。
- `log/` 只放运行期日志与回溯证据。
- persona 各自有独立 `config/`、`context/`，但共享同一套运行内核。

## 2. 角色总览

### 2.1 Matrix

- 默认入口。
- 是整个程序的主体、母体、总调度核。
- 负责理解用户、分流任务、协调系统、汇总结果。
- 持有全量工具能力的调度权，但不等于每一轮都亲自把所有事情做完。
- 可以直接工作，但默认更像总指挥、调度台、任务编排器。

### 2.2 Coding

- 专门处理编程、改代码、跑验证、读结构。
- 是施工副将，不负责长期陪聊和全局治理。

### 2.3 Companion

- 负责陪伴域、游戏路口、独立玩法上下文。
- 不承担主系统治理责任。

### 2.4 Advisor

- 只做内部审计与治理建议。
- 不直接对用户聊天，不直接接管主会话。

### 2.5 Department

- 负责日志、观测、回归证据、问题分诊。
- 是“行政政府”，不是业务模块。

## 3. Matrix 工作与职责定义

这一章是本轮优先落地的核心。

### 3.1 Matrix 的定位

`Matrix` 是 ProjectYing 的默认前台、主体母体与总调度者。

它既可以直接工作，也可以把自己“分裂”为更适合的执行单元。

它的核心职责不是“亲自吞掉所有执行细节”，而是：

- 接住用户当前目标。
- 判断这件事属于哪一类工作、哪一个执行层更合适。
- 选择最合适的工具链、skills、子 agent 或副将去推进。
- 在推进过程中保持用户可见、流程可解释、结果可交付。

一句话定义：

> Matrix 是母体与调度核：掌管全能力，但默认以编排、协调、分裂执行为主。

### 3.2 Matrix 必须负责的事情

#### A. 用户入口

- 接收用户问题、需求、任务、反馈。
- 判断当前是在闲聊、问答、开发、排障、测试、对账还是跨域协调。
- 在方向模糊时，先缩小问题，再推进执行。

#### B. 任务分流

- 判断是否继续留在 Matrix 内直接处理。
- 判断是否应切到 `Coding`、进入 `Companion` 路口，或触发专门的子 agent 流。
- 判断是否使用某个 skill，而不是裸调工具。
- 判断是否需要等待 `Advisor` 的治理结果。

#### C. 可见协同

- 负责把工具执行过程、阶段状态、最终结果汇总给用户。
- 负责让用户看懂“现在系统在做什么”，而不是只看到一坨结果。
- 负责在工具结束但正文为空时补可见收尾。

#### D. 工具调度与能力编排

- 掌管全量工具目录与能力注册表。
- 决定当前轮次向模型暴露哪些工具能力、哪些只作为内部调度能力存在。
- 决定何时直接调工具、何时走 skill、何时派生子 agent。

#### E. 轻量调查与直接执行

- 允许做轻量读取、检索、状态确认、截图查看、环境检查。
- 允许做少量、清晰、低风险的系统级操作。
- 允许在简单任务、快速修正、兜底情形下直接施工。
- 对需要长时间观察的任务，负责决定是否启用 PTY/后台终端。

#### F. 启动维护与系统状态汇报

- 新会话启动时，Matrix 负责接收启动维护票据。
- 负责判断上下文是否健康、系统状态是否异常。
- 正常则简短略过，异常则明确提醒用户。

#### G. 跨供应商通用协调

- 不以某一家 provider 的 schema 或 API 结构作为系统本体。
- 负责在统一能力模型之上，选择当前 provider 的最佳投影方式。
- 当 provider 不支持结构化工具调用时，仍要能退化到文本协议或半结构化协议继续工作。

### 3.3 Matrix 不默认亲自承担的事情

这是后续工具和目录收口的工作边界。注意：这里不是“完全不允许”，而是“不默认让 Matrix 亲自长期背着跑”。

#### A. 不默认亲自做长期治理

- `Matrix` 不应长期把 `write / summary / compact / add / replace` 当成主要工作。
- 表层上下文治理由系统维护链与 `Advisor` 负责。
- Matrix 在必要时可以直接治理，但默认应通过治理链和系统票据完成。

#### B. 不默认亲自做重型编程施工

- 大段代码编辑、成体系调试、长链修复，默认转交 `Coding` 或专门 agent。
- Matrix 保留直接施工权，但直接施工应该是特例、短链、兜底，而不是主工作形态。

#### C. 不和单一 provider 绑定

- Matrix 的能力定义不能写死在 Codex 风格 schema 里。
- DeepSeek、Gemini、Codex、其它兼容 Responses/Chat 的 provider，都应共享一套母体能力模型。
- provider 差异只允许落在适配层，不允许反向污染系统主体设计。

#### D. 不做隐式自维护循环

- Matrix 不应频繁为了“整理自己”而吞掉大量 token。
- 内部治理必须通过明确票据、明确触发条件、明确结果回执执行。

### 3.4 Matrix 的标准输入

Matrix 当前允许接收的输入，后续统一归到这六类：

1. **用户输入**
   - 聊天、提问、需求、测试反馈、对账回答。
2. **系统输入**
   - 启动维护、阈值提醒、失败重试、状态提醒。
3. **治理输入**
   - `Advisor` 的审计结果、治理建议、写回完成通知。
4. **执行输入**
   - 工具回执、PTY 快照、agent 结果、命令结果。
5. **记忆输入**
   - `memory_check / memory_read` 的定位结果与原文结果。
6. **技能输入**
   - skill 选择结果、skill 运行阶段状态、skill 执行回执。

### 3.5 Matrix 的标准输出

Matrix 对外只应该输出五类东西：

1. **用户可见回复**
   - 直接回答、阶段汇报、结果汇总、异常说明。
2. **执行请求**
   - 工具调用、只读调查、轻量操作、必要的终端任务。
3. **协同请求**
   - 对账、计划、任务模式、等待副将或治理结果。
4. **路由决策**
   - 建议切换到 `Coding` / `Companion`，或说明当前仍由 Matrix 继续处理。
5. **能力编排**
   - 选择某个 skill、派生 agent、挂载 provider 适配投影。

### 3.6 Matrix 的工具边界

后续结构收口时，以“全量能力由 Matrix 掌管，但不是全都直塞给当前模型”作为准则。

#### A. Matrix 持有全量能力主权

- 命令执行：`exec_command / write_stdin / pty_list / pty_kill`
- 文件编辑：`apply_patch`
- 交互协同：`request_user_input / update_plan / task_mode`
- 上下文治理：`context_manage.*`
- 长期记忆：`memory_check / memory_read / memory_add / memory_replace`
- agent 编排：`spawn_agent / send_input / wait_agent / resume_agent / close_agent`
- 视觉输入：`view_image`
- 后续扩展：`adb / termux-api / web / plugin / game toolset / provider-specific tools`

#### B. 运行时不直接暴露“全量 schema”

- Matrix 的内部能力全集，不等于当前轮次暴露给模型的 schema 全集。
- 运行时应根据任务类型、persona、provider 能力、风险等级，裁出一个“当前可见工具切片”。
- 用户看到的是统一工作流；模型实际看到的是当前任务最需要的那部分能力投影。

#### C. 工具能力按三层组织

1. **内核能力**
   - 系统真正支持的原子动作。
   - 例如：读文件、运行命令、补丁编辑、开启 PTY、读记忆、写上下文。
2. **调度能力**
   - Matrix 用来组合内核能力的能力。
   - 例如：进入任务模式、派生 agent、发起对账、选择 skill。
3. **投影能力**
   - 针对具体 provider 暴露出去的 schema / function / 文本协议形式。
   - 例如：Codex function schema、Gemini function declaration、DeepSeek 的兼容结构或纯文本工具协议。

#### D. Matrix 的直接执行原则

- 简单任务：Matrix 可直接用工具完成。
- 中等复杂任务：Matrix 优先组合工具或走 skill。
- 长链复杂任务：Matrix 优先分裂为 `Coding` 或其它专项 agent，再负责监督与汇总。
- 兜底场景：若副将不可用、provider 不稳、工具路径单一，Matrix 仍可亲自接管。

### 3.7 Matrix 与目录的主责边界

#### Matrix 直接主责

- `config/Matrix/`
- `context/codex/`
- `src/main.rs` 中与 Matrix persona 相关的会话、启动维护、前台分流逻辑
- `src/ui.rs` 中 Matrix 前台显示规则

#### Matrix 只读或调用，不主责

- `memory/`
- `log/department/`
- `src/mcp.rs`
- `src/departmentrs.rs`
- `context/advisor/`
- `context/codexcoding/`
- `context/codexcompanion/`

### 3.8 Matrix 的标准工作流

后续希望 Matrix 固定按这条链工作：

1. 接收用户目标。
2. 判断任务类型。
3. 判断是否需要对账。
4. 判断是否需要计划与任务模式。
5. 判断应直调工具、走 skill、还是分裂 agent。
6. 裁出当前轮次需要暴露的能力切片。
7. 推进可见执行。
8. 汇总结果并交付用户。

### 3.9 本章落地后的直接整改方向

这不是本轮全部改完的意思，而是后续施工顺序：

1. 先建立统一能力注册表，不再把某个 provider 的 schema 当成工具本体。
2. 明确 `Matrix -> Coding`、`Matrix -> Advisor`、`Matrix -> Companion` 的交接规范。
3. 建立“当前任务能力切片”机制，而不是把全量 schema 每轮硬塞给模型。
4. 把 skill 层插到 Matrix 与工具层之间，让 Matrix 先调度 skill，再调度原子工具。
5. 清理“逻辑在 prompt、schema、provider 适配里重复描述”的残留结构。

## 4. 工具母体与通用适配蓝图

这一章定义“工具如何作为系统能力存在”，而不是“某个 schema 文件长什么样”。

### 4.1 工具本体不是 schema

- `schema` 只是某种 provider 能理解的外壳。
- 真正的工具本体应是 ProjectYing 内部的“能力注册表”。
- `tool prompt` 也不是工具本体；它只是给模型看的可见说明层。
- 注册表至少定义：
  - 工具名与内部能力 ID
  - 输入结构
  - 输出结构
  - 风险等级
  - 是否可并行
  - 是否可给 Matrix 直用
  - 是否只允许经由 skill / agent 调度
  - 对不同 provider 的投影方式

### 4.1.1 单向编译原则

- 工具真源只保留一份：`Capability Registry + Toolbox State`。
- `tool prompt`、`schema`、`function declarations`、文本工具协议，都只能由真源单向生成。
- 不允许把 `schema` 或 provider-specific tool JSON 反过来当主仓编辑。
- 不允许让 `tool prompt` 和 `schema` 互相双向同步；否则必然漂移。

一句话：

> 工具卡片是可见层，schema 是投影产物，真源只能是内部能力注册表。

### 4.1.2 工具说明分两层

后续每个工具都只保留两种可见状态：

1. **折叠**
   - 显示工具名、功能摘要、适用场景。
2. **展开**
   - 显示调用格式、参数说明、使用说明、注意事项。

不再让模型直接阅读 provider 原生 JSON schema 作为主要工作材料。

### 4.2 统一能力模型

后续工具先按统一能力域分组：

1. **Read**
   - 文件读取、目录读取、图片读取、日志读取、记忆读取。
2. **Run**
   - 一次性命令、PTY、后台终端、状态查询。
3. **Edit**
   - `apply_patch`、后续可能的重写类编辑工具。
4. **Manage**
   - 计划、任务模式、上下文治理、记忆治理。
5. **Delegate**
   - 子 agent 生命周期、批量派发、结果回收。
6. **Consult**
   - 对账、用户选择、后续插件/问卷/调查能力。

统一能力模型是母体语言，provider schema 只是翻译器。

### 4.2.1 工具调度管理器

Matrix 后续通过统一的 `toolbox_manage` 管理工具可见性，而不是直接编辑工具定义。

建议动作固定为：

- `list`
  - 查看当前工具箱状态。
- `open`
  - 打开某个工具或工具组，进入展开态。
- `close`
  - 关闭某个工具或工具组，回到折叠态。
- `pin`
  - 固定常驻，不随任务自动收起。
- `unpin`
  - 解除常驻。

说明：

- 优先按稳定 `tool_id` 或 `group_id` 管理，不使用易漂移的数字编号。
- 若未来需要给用户做视觉编号，那也只作为 UI 显示编号，不作为内部主键。

### 4.2.2 工具箱状态

后续 `Toolbox State` 建议至少包含：

- `closed`
- `expanded`
- `pinned`

其中：

- `closed` 仍保留摘要，方便 Matrix 知道这工具存在。
- `expanded` 表示本轮上下文中展开完整使用说明。
- `pinned` 表示跨轮次常驻展开，直到明确取消。

### 4.3 Provider 适配层

后续 provider 统一分三层：

1. **能力层**
   - ProjectYing 内部标准工具与消息结构。
2. **适配层**
   - 把能力层投影为某个 provider 支持的协议。
3. **传输层**
   - 真正发 HTTP / SSE / responses / chat / function call 的地方。

### 4.3.1 投影规则

发送前统一按这条链路生成：

1. `Capability Registry`
2. `Toolbox State`
3. 生成当前轮次 `tool prompt`
4. 按 provider 能力决定是否额外生成：
   - `schema / tools`
   - `function declarations`
   - `MCP bridge`
   - `text tool protocol`

也就是：

- 通用母体上下文里永远可以有 `tool prompt`
- provider 若支持原生工具，再额外挂载结构化工具投影
- provider 若不支持原生工具，就只保留 `tool prompt + 文本协议`

### 4.4 目标支持面

当前蓝图默认覆盖：

- `Codex / Responses` 型
- `OpenAI-compatible chat/responses` 型
- `DeepSeek` 兼容型
- `Gemini CLI / Gemini function declaration` 型
- 纯文本工具协议兜底型

### 4.5 通用投影策略

不同 provider 下，能力投影按这套优先级：

1. **原生结构化工具调用**
   - 最优先。
2. **兼容函数调用**
   - 次优先。
3. **文本协议工具调用**
   - 当 provider 不支持 schema 或支持很差时使用。
4. **半自动回退**
   - 实在不支持结构化时，至少保留可读可解析的文本命令协议。

### 4.6 通用模式要求

- 上层 prompt 不应写死某家 provider 的术语。
- 大部分工具说明应来自内部能力注册表，不应在每家 provider 重写一遍业务语义。
- 适配层只改“小差异”：字段名、消息封装、工具投影、流式解析。
- 系统逻辑、工具语义、记忆结构、上下文工程保持统一。
- 若 provider 支持原生工具，则 `tool prompt` 与结构化工具投影应同时来自同一份真源，但不互相双写。
- 若 provider 只支持 `system / user / assistant`，则依然保留工具箱机制，只是退化为文本工具协议路线。

## 5. Skills 层蓝图

`skills` 是插在 Matrix 与原子工具之间的中层能力。

### 5.1 Skills 的定位

- skill 不是单个工具。
- skill 也不是整个人格。
- skill 是一组“可复用的任务套路、工具编排、规则约束、结果契约”。

Matrix 不应每次都从零拼一条工具链，而应优先选择 skill。

### 5.2 Skills 的作用

- 给 Matrix 一个更稳定的调度单位。
- 把复杂的多工具套路收口成可复用资产。
- 降低不同 provider 下直接暴露全量工具的压力。
- 让 Matrix 更像母体和总调度者，而不是手动操作员。

### 5.3 Skills 的建议结构

每个 skill 后续至少应定义：

- `skill_id`
- 目标领域
- 适用 persona
- 允许调用的工具子集
- 推荐的执行顺序
- 输入要求
- 输出要求
- 是否允许派生 agent
- 是否需要用户对账
- 对 provider 的特殊注意事项

### 5.4 第一批适合建立的 skills

1. **coding_repair**
   - 读代码、定位、补丁、验证、汇报。
2. **ui_regression**
   - 读日志、看截图、核对 UI 事件、给修复方向。
3. **context_audit**
   - 治理上下文、核对 AdviceBoard、生成治理建议。
4. **memory_trace**
   - 先 check，再 read；必要时回溯日期与 entry。
5. **provider_probe**
   - 检测当前 provider/API 能力、结构支持、失败类型。

### 5.5 Skills 与 Matrix 的关系

- Matrix 决定选哪个 skill。
- skill 决定使用哪些工具组合。
- 工具真正完成原子执行。
- provider 适配层只负责把 skill 需要的能力切片投影出去。

## 6. Matrix 上下文工程蓝图

既然 Matrix 是母体，就不能只靠“整段 prompt + 整份 schema”硬喂。

### 6.1 Matrix 上下文的目标

- 让 Matrix 始终知道自己是谁、当前要做什么、可调度什么。
- 让 Matrix 看到的是“当前任务最有用的能力摘要”，不是全量噪音。
- 让不同 provider 都能吃下同一套母体语义。

### 6.2 建议的上下文分层

后续建议 Matrix 实际收到的结构固定为：

1. **toolbox**
   - 当前已折叠/展开/常驻的工具卡片。
2. **charter**
   - Matrix 身份、职责、总规则。
3. **mission**
   - 当前用户目标、当前轮任务、当前计划、当前 task mode。
4. **skills**
   - 当前轮可调度的 skill 与套路摘要。
5. **memory**
   - `fastmemory`、必要的长期记忆定位结果。
6. **advice board**
   - 与 `Advisor` 的交流板。
7. **fastcontext**
   - 高层压缩上下文。
8. **context**
   - 当前工作上下文。

发送到具体 provider 前，再额外生成一层不对母体公开的：

9. **provider projection**
   - `schema / tools / function declarations / MCP / text tool protocol`

### 6.3 不再推荐的做法

- 不再把“全量 schema + 全量 prompt + 全量上下文”每轮硬拼给模型。
- 不再让 provider 适配层反过来决定 Matrix 的人格与职责。
- 不再把某些工具规则同时写三遍：prompt 一遍、schema 一遍、代码注释一遍。
- 不再把 provider 原生 schema 当作可长期维护的业务真源。

## 8. 当前状态说明

本文件先定义“目标边界”，不代表代码已经完全对齐。

当前至少还有两类已知偏差：

- 当前工具本体仍主要体现在 schema 文件与执行代码里，尚未抽成统一能力注册表。

- Matrix 现有 prompt / schema / provider 适配仍有职责重复和术语重复。
- skill 层目前还没有成为正式中层。
- 各 persona 的 prompt / schema / 目录边界还没有全部收紧到蓝图要求。

后续每一轮结构整改，都应以“让代码更接近本蓝图”为准，而不是继续堆功能。
